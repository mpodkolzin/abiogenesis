# Abiogenesis - Design Document

## Overview

Abiogenesis is an interactive 2D particle life simulator built with **Rust** and the **Bevy 0.16** game engine. It models emergent behavior from simple pairwise interaction rules between colored particle species - inspired by [Lenia](https://en.wikipedia.org/wiki/Lenia) and particle life simulations. Users observe, tune, and manipulate the simulation in real-time via a polished UI.

The simulation runs in a **toroidal universe** (edges wrap) with up to **3,000 particles** of **6 color species**. Each species pair has a configurable attraction/repulsion weight, forming an N×N interaction matrix that drives all emergent behavior.

**Platforms:** Native (macOS/Linux/Windows) and WebAssembly (WebGL2).

## Running & Deployment

### Native

```bash
# Just run it — opens a window with the simulation
cargo run --no-default-features

# With egui inspector (development)
cargo run
```

No server, no config. Opens a window directly.

### WASM — Local Testing

```bash
# 1. Build + generate bindings (see WASM Build above)
# 2. Serve the site/ directory with any static file server:
simple-http-server site/
# or:
python3 -m http.server -d site/
```

### WASM — Deployment

The `site/` directory is the deployment root, ready for any static host:

```
site/
├── index.html         ← Loading screen, audio context, clipboard import/export
├── assets/            ← Symlink to abiogenesis/assets/
├── index.js           ← Generated by wasm-bindgen (not committed)
├── index_bg.wasm      ← Generated by wasm-bindgen (not committed)
└── .gitignore         ← Ignores generated files, keeps index.html + assets
```

Only `index.html` and the `assets` symlink are committed. The JS and WASM files are generated by the build pipeline.

Deploy to any static host: GitHub Pages, Netlify, Cloudflare Pages, itch.io (this was a **Bevy Jam #6** entry).

### State Sharing (WASM only)

The web version uses the **clipboard** for state transfer — no backend required:

- **Bestow** (export): Serializes `SimulationParams + Model` as JSON, copies to clipboard
- **Receive** (import): Reads JSON from clipboard, applies to simulation

On native, both operations log a warning ("not supported on native").

```
main.rs
  ├── bevy_systems()       → DefaultPlugins, window config, clear color
  ├── third_party_systems() → TweeningPlugin, egui (dev), hot reload (dev)
  └── app_systems()        → Application plugins + system ordering
        ├── ParticlePlugin     (simulation core)
        ├── UIPlugin           (all UI)
        ├── ScenePlugin        (placeholder)
        ├── CameraPlugin       (2D orthographic camera)
        ├── ControlsPlugin     (mouse/touch input)
        └── BrowserStatePlugin (WASM state sync)
```

### System Ordering (Update schedule)

```
TickTimers → AnimationUpdate → RecordInput → Update → Camera
```

Defined via `AppSystems` enum in `systems.rs`. The chain ensures input is processed before physics, and camera follows after positions are updated.

## Module Reference

### `particles/` — Simulation Core

#### `particle.rs` — Entity Definition

- **`Particle`** — Marker component. Immutable, with `on_add`/`on_remove` hooks that maintain `ParticleIndex`.
- **`ParticleIndex`** — `Resource(Vec<Entity>)`. Central registry of all live particles. Capped at `MAX_PARTICLES = 3000`.
- **`Velocity`** — `Component(Vec2)`. Per-particle velocity, `Deref/DerefMut` to `Vec2`.
- **Required components:** `Transform`, `ParticleColour`, `Velocity` (via `#[require(...)]`).

#### `colour.rs` — Species System

Six species as an enum:

| Variant | Index | RGB |
|---------|-------|-----|
| Red     | 0     | (172, 40, 71) |
| Green   | 1     | (90, 181, 82) |
| Blue    | 2     | (51, 136, 222) |
| Orange  | 3     | (255, 155, 37) |
| Pink    | 4     | (233, 75, 234) |
| Aqua    | 5     | (57, 247, 241) |

- `ParticleColour` is an immutable component with an `on_insert` hook that auto-sets the correct `MeshMaterial2d<ColorMaterial>` from `ParticleAssets`.
- Key methods: `index()`, `from_index(usize)`, `random(particle_variety)`.
- `NUM_COLOURS = 6` is the maximum; `SimulationParams::num_colours` controls how many are active.

#### `model.rs` — Interaction Matrix

- **`Model`** — `Resource(Vec<f32>)`. Flat `NUM_COLOURS × NUM_COLOURS` array of interaction weights in `[-1.0, 1.0]`.
  - `weight(source, target) → f32` — lookup.
  - `set_weight(source, target, value)` — mutation.
  - `from_3x3` / `from_6x6` — constructors that zero-pad to full 6×6.
  - Serialized as `Vec<i32>` (weights × 100) for compact URL encoding.

- **`PRESETS`** — `LazyLock` array of 9 named presets: `(name, Model, SpawnerConfig, SimulationParams)`.

| Preset | Colors | Character |
|--------|--------|-----------|
| The First Garden | 3 | Balanced symbiosis |
| Circle of Life | 3 | Cyclic predation |
| Jörmungandr | 3 | Self-repelling chains |
| Rainbow Serpent | 6 | Extended chain topology |
| Predation | 3 | Asymmetric predator-prey |
| Ouroboros | 6 | Cyclic self-attraction |
| The Trinity | 3 | High friction, large radius |
| Divine Engine | 3 | Rotational dynamics |
| Heat Death | 3 | Zero interactions (entropy) |

- **Events:**
  - `Randomise` — Randomizes model weights and simulation params using biased normal distributions (favoring interesting results).
  - `ClearParticles` — Despawns all particles, sets decay to 0 (design mode).

#### `simulation.rs` — Physics Engine

**`SimulationParams`** — The 7 tunable knobs:

| Parameter | Range | Default | Purpose |
|-----------|-------|---------|---------|
| `friction` | 0–5 | 2.0 | Exponential velocity damping |
| `force_strength` | 0–200 | 100.0 | Force magnitude multiplier |
| `attraction_radius` | 0–200 | 75.0 | Max interaction distance |
| `peak_attraction_radius` | 0–200 | 50.0 | Distance of peak attraction |
| `repulsion_radius` | 0–200 | 25.0 | Boundary between repulsion and attraction |
| `decay_rate` | 0–200 | 100.0 | Particles recycled per second |
| `num_colours` | 1–6 | 6 | Active species count |

**Force model** (`magnitude` function):

```
distance:  0 ──── repulsion_r ──── peak_attraction_r ──── attraction_r
force:    -1 ──── 0 ──────────── factor ──────────────── 0
              repulsion zone     attraction ramp-up       attraction decay
```

- All transitions use `remap()` (linear interpolation).
- `factor` is the model weight for the (source, target) color pair.
- Negative factor = repulsion in the attraction zone.

**`compute_forces` system:**
1. Rebuild spatial index every frame.
2. For each particle (parallel via `par_iter_mut` on non-hot-reload):
   - Query neighbors within `attraction_radius`.
   - Sum forces from `magnitude × force_strength × normalized_displacement`.
   - Apply velocity: `v += force × dt`, then `v *= e^(-friction × dt)`, clamp to 200.
   - Wrap position toroidally.

#### `spawner.rs` — Lifecycle Management

- **`SpawnerConfig`** — `None` | `Uniform` (default) | `Custom(Vec<(colour, shape)>)`.
- **`SpawnShape`** — `Rect` | `Circle` | `HollowCircle`. Generates random positions within shape.
- **`Respawn` event** — Despawns all particles, respawns `MAX_PARTICLES` with even color distribution.
- **`SpawnParticle` event** — Spawns one particle at position. If at capacity, recycles the oldest (FIFO via `OldestParticle` index).
- **`update_colours_on_num_change`** — When `num_colours` changes, randomly reassigns all particle colors.
- **`ParticleAssets`** — Shared mesh handle (unit circle) + 6 color material handles.

#### `spatial_index.rs` — Neighbor Queries

Thin wrapper around `SpatialHashGrid<(Entity, ParticleColour)>`:
- Initialized at startup: 19×10 grid covering simulation bounds.
- Resized on window resize events.

#### `decay.rs` — Population Recycling

Runs in `FixedUpdate` at 0.1s intervals:
- Respawns `decay_rate × 0.1` oldest particles per tick at random positions with random colors.
- Skips the followed particle (if any).
- Creates a "steady state" where old structures dissolve and new ones form.

#### `size.rs` — Simulation Viewport

- `SimulationSize` SystemParam — Derives dimensions from window size, clamped to minimum 1920×1080.
- `scale_bounds()` — Returns `(0.1, max_zoom)` where max_zoom prevents seeing past simulation edges.

### `spatial_hash.rs` — Generic Spatial Hash Grid

Standalone, generic data structure:
- `SpatialHashGrid<T>` — Grid of `Vec<(Vec2, T)>` cells.
- **Toroidal queries:** `get_query_cells` uses `gen` blocks (coroutines) to yield wrapped cell coordinates.
- **API:** `insert(pos, item)`, `query(pos, radius) → Iterator`, `clear()`, `update_bounds()`.
- 13 unit tests covering wrapping, multi-cell queries, radius filtering.

### `math.rs` — Numeric Utilities

- `lerp(a, b, t)`, `inverse_lerp(a, b, value)`, `remap(value, a, b, c, d)`.
- **`TorodialMath` trait** on `Rect`:
  - `toroidal_displacement(a, b)` — Shortest displacement vector accounting for wrap.
  - `toroidal_wrap(pos)` — Clamp position into bounds via wrapping.
- 14 unit tests.

### `camera.rs` — Camera System

- 2D orthographic camera.
- **Zoom clamping:** Enforced every frame via `scale_bounds()`.
- **Follow mode:** `FollowParticle(Entity)` resource. When active, smoothly translates *all particles* to keep the target centered (rather than moving the camera). Removed if entity despawns.
- **`Viewport` SystemParam** — Provides `to_world(screen_pos) → Vec2` conversion.

### `controls.rs` — Input Handling

All input handlers are entity observers attached to the full-screen UI container:

| Handler | Trigger | Action |
|---------|---------|--------|
| `drag_screen` | Right-drag | Pan (translates all particles) |
| `scroll_wheel_zoom` | Scroll | Zoom camera |
| `select_follow_particle` | Left-click (Camera tool) | Follow nearest particle |
| `particle_brush_start/drag` | Left-click/drag (Particle tool) | Spawn particles |
| `eraser_brush_start/drag` | Left-click/drag (Smite tool) | Despawn particles in 30px radius |
| `touch_pan` | Two-finger gesture | Pan + pinch zoom + rotation |

**Design note:** Pan/zoom works by moving all particles rather than the camera. This keeps the camera at the origin and simplifies coordinate math.

### `browser_state.rs` — WASM State Persistence

- **WASM:** Serializes `SimulationParams + Model` as `State`, passes to/from JavaScript via `wasm-bindgen`. Uses a thread-local pending import cell polled each frame.
- **Native:** Stub implementations that log warnings.
- **Events:** `Import`, `Export`.

- `LensPlugin` — Custom tweening lenses.

## Data Flow

### Simulation Loop (per frame)

```
1. RecordInput
   ├── Touch gestures → pan/zoom
   ├── Window resize → update spatial index bounds
   └── WASM → poll pending import

2. Update
   ├── Rebuild SpatialIndex from all particle positions
   ├── For each particle (parallel):
   │   ├── Query neighbors within attraction_radius
   │   ├── Sum pairwise forces (magnitude × weight × direction)
   │   ├── velocity += force × dt
   │   ├── velocity *= friction_damping
   │   └── position = toroidal_wrap(position + velocity × dt)
   └── UI: sync model matrix display, slider positions

3. Camera
   ├── Clamp zoom to scale_bounds
   └── If following: translate all particles to center target

4. FixedUpdate (0.1s)
   └── Decay: respawn oldest N particles at random positions
```

### Event-Driven Actions

```
User clicks "Reshape"  → triggers Randomise event
                        → randomise_model observer mutates Model + SimulationParams

User clicks "Regenerate" → triggers Respawn event
                         → respawn_particles observer despawns all, spawns MAX_PARTICLES

User clicks "Annihilate" → triggers ClearParticles event
                         → clear_particles observer despawns all, sets decay_rate = 0

User selects preset     → directly inserts Model, SimulationParams, SpawnerConfig resources
                        → triggers Respawn

User drags model box    → directly mutates Model resource weights
User drags slider       → directly mutates SimulationParams resource fields
```

## Key Design Decisions

1. **Particles move, camera stays.** Pan/zoom is implemented by translating all particle positions, not the camera. This simplifies viewport math and the follow-particle system.

2. **Component hooks for indexing.** `Particle::on_add/on_remove` automatically maintain `ParticleIndex`, avoiding manual bookkeeping or system ordering issues.

3. **Immutable marker components.** `Particle` and `ParticleColour` are `#[component(immutable)]`, preventing accidental mutation and enabling hook-based side effects on insert.

4. **Spatial hashing with toroidal wrapping.** The 19×10 grid wraps at boundaries, so particles near edges correctly detect neighbors across the wrap. Uses Rust's `gen` blocks for ergonomic cell iteration.

5. **Decay as population control.** Rather than hard-capping spawns, the decay system continuously recycles the oldest particles. This creates a dynamic steady state where structures form, age, and dissolve.

6. **Observer pattern for one-shot actions.** `Randomise`, `ClearParticles`, `Respawn`, `Export`, `Import` are all handled by Bevy observers rather than polling systems.

7. **Hot reload support.** All physics and UI systems are annotated with `#[hot]` behind a feature flag, enabling live iteration during development.

8. **Parallel force computation.** `par_iter_mut` is used in release mode for the force loop, falling back to serial iteration when hot reload is active (due to borrowing constraints).
